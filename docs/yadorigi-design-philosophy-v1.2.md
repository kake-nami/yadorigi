## プロダクト名と由来

**Yadorigi（宿り木）**

宿り木は、他の樹に宿って育つ植物。鳥が運んだ種子が枝に芽吹く。このプロダクトは、Xという大きな樹のうえで、ユーザーの手元に独立して育つ宿り木。

- Xの青い鳥が運んでくるtweet = 種子
- ローカルに育つこのプロダクト = 宿り木
- 必要なとき、宿り木はユーザーの行動に結びつく

「**Xに溶け込み、Xから一歩離れる**」という大方針は、この命名にも反映されている。**依存しているが寄生していない。共生しているが独立している**。

### 姉妹プロダクト Tsuzuri との narrative

> **「Yadorigi で集めて、Tsuzuri で綴る」**

| 製品 | 役割 |
|---|---|
| **Yadorigi（宿り木）** | 入力 — 読む、蓄える、再訪する |
| **Tsuzuri（綴り）** | 出力 — 綴る、編む、公開する |

知的生産の入り口と出口を、同じ哲学コードを共有するローカルOSSで支える。

## 0. メタ原則：考えさせないことが、最大のリスペクトである ★v1.2新設

> **このプロダクトはユーザーの思考を奪わない。代わりに、ユーザーの思考のために考えさせない設計を追求する。**

### 4側面の徹底

| 領域 | 原則 |
|---|---|
| **同意** | 1クリックで進める |
| **クラスタ** | 説明不要、すでにあるように見せる |
| **計測** | 裏でやる、UIには出さない |
| **暗号化** | 意識させない（ただしリカバリーは明示） |

### 成功と失敗の判定

- ユーザーが Yadorigi のことを考える時間が長いほど、Yadorigi は失敗している
- ユーザーが自分の思考に没頭できているとき、Yadorigi は成功している

### 他原則との関係

このメタ原則は、規範性宣言3「約束は感情で煽らない」と4「約束は守られなくても価値を生む」を統合する**最上位制約**。下位のすべての原則（トーン、装飾密度、AI as Accelerator、計測哲学、クラスタリングUI原則）はこのメタ原則の具体化として位置づけられる。

### 由来

このメタ原則は、ChatGPT 第3ラウンドレビューの総括メッセージ「**正しい設計を作ろうとして、雑に使う現実をまだ織り込めていない**」を内部化したもの。設計者の正しさより、ユーザーの怠惰さを尊重する姿勢への転換。

## 1. 大方針

> **「Xに溶け込み、Xから一歩離れる」**

X UIの密度・タイポグラフィ・色彩制限を継承しつつ、Xにはない**思考の余白**（Kanbanの間、ActionLinkの構造）を加える。**Xから来た人が違和感なく使い始められ、Xから抜け出せる体験**を作る。

### 規範性の宣言

> **このプロダクトは、「保存することは、未来の自分との約束である」という規範を提示する。**

派生する4つのサブ規範：

1. **約束は計算可能である**
   - 「読む／行動する」と決めた瞬間、それは状態を持つ
   - Kanbanはその状態化の道具

2. **約束は感情で煽られない**
   - 未読件数で焦らせない
   - ストリークで束縛しない
   - Xにはない「呼吸の許される場所」

3. **約束は移動可能である**
   - データはローカル所有
   - Export可能
   - いつでも他のツールへ持ち出せることが約束履行の前提

4. **約束は守られなくても価値を生む（v1.2追加）**
   - ユーザーがactionLinkを作らなくても、tag付けしなくても、整理しなくても
   - ルールベースクラスタが暗黙の構造を作り出す
   - 自動の再発見が、忘れられた約束を蘇らせる
   - 怠惰さは尊重される。価値はその上に立つ

これは**Xを否定する**設計ではない。Xに**蓄積された約束を、Xの中では果たせない**という構造的問題に応答する設計。

## 2. トーン規定

### 2.1 コピー方針：中立 + 事実報告寄り

Xのコピー文化（dry / wry / 短文 / 主語省略）を継承する。

| 場面 | NG表現 | OK表現 |
|---|---|---|
| 未整理通知 | 「ブックマークが整理されていません！」 | 「未分類: 12件」 |
| 完了フィードバック | 「素晴らしい！32件を整理しました🎉」 | 「32件を整理しました」 |
| 削除確認 | 「本当に削除しますか？取り消せません！」 | 「削除しますか？」 |
| AIサジェスト | 「AIがあなたのために最適なカテゴリを発見しました！」 | 「提案: TO_READ → IN_PROGRESS」 |
| エラー | 「申し訳ありません、エラーが発生しました😢」 | 「同期できませんでした。再試行」 |
| 空状態 | 「まだ何もありません。さあ始めましょう！」 | 「TO_READ は空です」 |
| 初回オンボーディング | 「ようこそ！Yadorigi へ！」 | 「Yadorigi へようこそ」 |
| インポート完了 | 「やった！1,234件のbookmarkを取り込みました！」 | 「1,234件を取り込みました」 |
| **同意フロー（v1.2追加）** | 「あなたの体験向上のため、ブラウザ履歴へのアクセスを許可してください。これにより…」 | 「過去の閲覧から"再発見"を作りますか？[はい（推奨）] [いいえ]」 |

### 2.2 警告表現：控えめ

- 赤色フラッシュ・全画面モーダル・「！」アイコン濫用は禁止
- インライン表示の小さなテキスト警告を基本とする
- **ただし「リカバリーキーの保存」のような不可逆な操作の警告は強調してよい**（v1.2追記）

### 2.3 多言語対応の注意点

- MVP / v1.x 段階：日本語のみ
- 将来 i18n 対応する際も、上記NG/OKルールは英語にも適用

## 5. AI as Accelerator ★v1.2リネーム（旧：AIサジェスト first 哲学）

### 5.1 基本原則

> **「AIは加速装置である。コア体験はAIなしで完結する」**

Yadorigi は AI なしで価値が成立するように設計されている。AI機能は速度と精度を加速するが、それなしでも：

- Kanban の状態管理は手動でフル機能
- ActionLink は手動で作成可能
- Entity抽出（hashtag/URL/mention/100+ ツール辞書）は AI なしで動作
- 全文検索とMindmapも AI なしで動作
- **暗黙クラスタリング2層（時間近接 + エンティティ）は AI なしで動作**（v1.2追加）

### 5.2 具体的なルール

- AIによる**全自動分類**禁止：必ず「提案 → 承認/却下」のステップを介す
- AI判断の**透明性**：「なぜこの提案？」をユーザーが見られる
- AI使用の**完全オプトイン**：default OFF、設定でオン
- ローカルLLM（Ollama）対応の優先度を高く保つ（v1.x）
- Anthropic / OpenAI API使用時は**越境移転であることを明示**

### 5.3 AIフィードバックループ

- accept/reject ログをローカル保存
- これがユーザーの判断履歴として育つ（System of Record性）
- ユーザーの判断にAIが追従する設計（逆ではない）

### 5.4 AI失敗時の振る舞い

- AI APIエラー時はEntity抽出 + 暗黙クラスタリングのみで動作継続
- リトライUIは控えめに
- ユーザーの作業を中断させない

## 7. 計測哲学：見せない計測 ★v1.2全面書き換え

> **「見せることが、信頼を毀損してはならない」**

### 7.1 基本原則

「測ると壊れる」は半分正しい。完全な真実は「**見せると壊れる**」である。
Yadorigi は裏で行動シグナルを記録するが、UIには一切表示しない。これにより Goodhart's Law を構造的に回避しつつ、改善ループを維持する。

### 7.2 ローカルで取得する3つの行動シグナル

すべて BehaviorLog テーブルに保存、UIには一切表示しない：

| # | シグナル | 計算方法 | 死活閾値 |
|---|---|---|---|
| 1 | **継続利用率** | 7日以内に2回以上起動した user 割合 | < 30% で価値発火失敗 |
| 2 | **データ残存率** | 作成された bookmark が7日後も残っている割合 | < 70% で価値なし |
| 3 | **再訪発生率（コアKPI）** | 作成後7日以内に再度開かれたbookmark数 ÷ 作成総数（bookmark単位×7日窓） | < 15% でコア体験届かず |

### 7.3 シグナル悪化対応マトリクス

| パターン | 対応 |
|---|---|
| 全部悪い | プロダクト成立せず → コンセプトから見直し → やらなかった集行きの可能性 |
| 再訪だけ悪い | コア価値崩壊 → 最優先で修正 |
| 継続だけ悪い | 初期体験 or 習慣設計の問題 |
| 残存だけ悪い | ノイズ多 or 保存価値低 |

### 7.4 サーベイの位置づけ

3-6ヶ月ごとのサーベイ（Tally / Google Forms）：
- **使える用途**：仮説生成、違和感の検知
- **使えない用途**：定量判断、優先順位決定
- N=30で判断し始めると普通に間違える、と認識する

### 7.5 v1.x：コア体験A発生時のXシェアボタン（タイプβのみ）

整理statsの共有は禁止（タイプα = ゲーミフィケーション）。

タイプβ：コア体験A発生時の自発共有（v1.1継承）。

### 7.6 v2：opt-in テレメトリ

default OFF、設定でオン。法人化後の実装。

### 7.7 Goodhart's Law の構造的回避

- 数字を直接共有する仕組みは導入しない
- タイプβは「価値を感じた瞬間にだけ共有する」設計
- メトリクスを手段として保ちつつ、絶対視しない
- **行動シグナルをUIに出さないことで、ユーザーが目標化できない**

## 9. 反証可能性

哲学コード自体への自己検証（v1.1継承 + v1.2追記）：

- **シグナルA**：「もっと派手にしてくれ」が多発 → 「中の下」密度の見直し
- **シグナルB**：コントリビューターが「この哲学きついから別フォーク」 → 哲学コード柔軟化を検討
- **シグナルC**：design token化のコストが想定の3倍 → リファクタ判断
- **シグナルD**：「測らない哲学」がコミュニティに浸透せず → opt-in テレメトリの前倒し検討
- **シグナルE**：Xシェアボタンがほとんど押されない → コア体験定義 or 出現条件の見直し
- **シグナルF**：Xシェアボタンがゲーミフィケーション化 → 出現条件厳格化、機能停止
- **シグナルG（v1.2追加）**：暗黙クラスタリングのノイズが多すぎてユーザー混乱 → MVP段階での即修正、行動ベース永久延期検討
- **シグナルH（v1.2追加）**：DB暗号化のリカバリーキー喪失でデータロス報告が複数発生 → リカバリー設計の強化、強制バックアップ機能検討
- **シグナルI（v1.2追加）**：行動シグナル「再訪発生率 < 15%」が継続 → コア体験崩壊、Step全体の見直し

## 11. クラスタリングUI原則 ★v1.2新設

暗黙クラスタリングのUI設計に関する固有の原則。第0章メタ原則の具体化として位置づけ。

### 11.1 「分類を強制しない、すでにそこにあるように見せる」

ルールベースクラスタの結果は、ユーザーへの問いかけとしてではなく、**観察結果として表示**する。

| シーン | NG | OK |
|---|---|---|
| 関連発見 | 「これらをまとめますか？」 | bookmarkカードに「他に3件、似た内容」と小さくラベル表示 |
| クラスタ生成 | 「クラスタを作成しました」モーダル | mindmap上で勝手にクラスタが見えるだけ |
| 提案表示 | 「以下の3件を1つのプロジェクトに紐付けしましょう」 | 「似た文脈の3件」として軽く表示、クリックで詳細 |
| 学習通知 | 「学習中：あなたの好みを分析しています」 | 何も表示しない（裏で動作） |

### 11.2 「間違ってても害がない設計」原則

ルールベースは精度が低くノイズが出る前提：

- 自動フォルダ化はしない（提案レベルに留める）
- 「合ってる時だけ意味を持ち、外れた時は無視できる」UI
- ユーザー側で1クリックで dismiss可能
- dismiss → 内部的に重み下げ、ユーザーには「学習している」と見せない
- 手動結合 → 内部的に重み上げ

### 11.3 MVP実装範囲

採用：
- **時間近接クラスタ**（24h窓 + カテゴリ類似、最も安定）
- **共通エンティティクラスタ**（URL domain / 人物名 / TF-IDF上位語、補助）

未採用（v2以降）：
- **行動ベースクラスタ**（最もノイズ、解釈困難）

### 11.4 学習はしている、しかしユーザーに気づかせない

- dismiss / 手動結合のフィードバックを内部的に活用
- 重み調整によりクラスタリング精度を改善
- ただし「学習中」「精度向上中」のようなUIは出さない
- ユーザーから見れば「最初から良くなっていく」だけ

### 11.5 由来

このセクションは ChatGPT 第2/第3ラウンドレビュー Q2 を反映したもの。「**UIをミスると終わる**」という指摘を、Yadorigi 哲学コードの一部として恒久的に組み込む。
